# Bench d'extraction — modèles Ollama locaux

Date : 2026-07-03 · Pipeline : minerva `9eaa3d0` · Résultats bruts : [`results/2026-07-03/`](results/2026-07-03/)

## Objectif

Comparer les modèles disponibles sur l'Ollama local pour la tâche cible de
minerva : extraire d'un texte littéraire les entités nommées, leurs attributs
dynamiques et les relations entre entités, via sortie structurée (JSON Schema).
Le but est de choisir le modèle à utiliser pour traiter un roman entier, et
d'identifier les limites du pipeline lui-même.

## Environnement

| | |
|---|---|
| Machine | Mac Studio, Apple M3 Ultra, 96 Go de RAM unifiée |
| Serveur | Ollama 0.31.1, endpoint compatible OpenAI (`http://localhost:11434/v1`) |
| Pipeline | `minerva extract` (backend `OpenAIBackend`, `response_format` JSON Schema) |
| Modèles testés | `qwen2.5:7b-instruct`, `gemma4:latest`, `qwen3.6:latest`, `gpt-oss:20b`, `mistral-small3.2:24b-instruct-2506-q8_0`, `gpt-oss:120b` |
| Exclu | `bge-m3` (modèle d'embedding, pas de génération) |

## Protocole

- **Texte** : [`bench_texte.txt`](bench_texte.txt) — résumé en 4 paragraphes du
  début des *Misérables* (1 624 caractères), choisi parce qu'il contient un
  piège de résolution d'identité explicite : « Nul ne savait que Madeleine
  n'était autre que Jean Valjean. »
- **Chunking** : `chunk_size=700` → 3 chunks, pour tester la cohérence
  inter-chunks (le prompt du chunk N+1 rappelle les entités déjà connues).
- **Chrono** : un appel d'échauffement charge le modèle en mémoire **hors
  chrono** (`load_s`) ; `time_s` mesure uniquement les 3 appels d'extraction.
- **Un seul run par modèle** — voir Limites.
- **Scoring automatique** (`bench.py`) contre une référence :
  - 11 entités attendues (personnages et lieux), résolues via le mécanisme
    d'alias du graphe, avec variantes de noms tolérées ;
  - 6 relations clés attendues — seule l'existence d'un lien entre les deux
    bonnes entités est exigée, le nom de la relation est libre ;
  - **test de fusion** : « M. Madeleine » se résout-il vers la même entité que
    « Jean Valjean » ? Ne réussit que si le modèle déclare l'alias, puisque la
    fusion côté pipeline est déterministe.

## Analyse par modèle

### gpt-oss:120b — le seul à tout réussir

Couverture parfaite (11/11 entités, 6/6 relations) et **seul modèle à
déclarer l'alias décisif**, produisant une entité unique et riche :

```json
{"name": "Jean Valjean", "type": "personnage", "aliases": ["M. Madeleine"],
 "attributes": {"profession": "forçat", "durée du bagne": "dix-neuf ans",
                "lieu du bagne": "Toulon", "passeport": "jaune",
                "poste": "maire de Montreuil-sur-Mer", "force": "hérculéenne"}}
```

Les relations portent des attributs pertinents (`arrive à Digne
{date: "un soir d'octobre"}`, `accueille {lieu: "maison de l'évêque"}`).

Défauts observés (utiles pour durcir le pipeline, voir Recommandations) :
une entité au nom vide, une relation sans cible (`s'enfuit`), un doublon
« Montreuil-sur-Mer » causé par un **tiret insécable U+2011** dans l'une des
deux graphies, et « inspecteur Javert » / « Javert » non fusionnés.

### qwen3.6:latest — excellent mais 2,7× plus lent que le 120b

Couverture parfaite des entités et relations, bons attributs, mais rate la
fusion d'alias. C'est un modèle « thinking » : il raisonne longuement avant de
répondre, d'où 267 s d'extraction — paradoxalement plus lent que le 120b
(architecture MoE, peu de paramètres actifs par token).

### gpt-oss:20b — le meilleur compromis vitesse/qualité

86 s, tous les personnages présents, 5/6 relations. Il ne manque que deux
lieux. Recommandé pour itérer sur un roman entier quand le 120b est trop lent.

### gemma4:latest — bonne couverture d'entités, relations plus faibles

Toutes les entités attendues, mais seulement 4/6 relations clés et pas de
fusion d'alias. Rapide (64 s).

### qwen2.5:7b-instruct — le plus rapide, pour prototyper

20 s d'extraction. Correct sur les personnages principaux, perd les entités
secondaires et la moitié des relations.

### mistral-small3.2:24b-q8 — contre-performance nette

Le pire du lot malgré ses 24 Md de paramètres en q8. Les 3 chunks ont bien été
appelés (vérifié dans le log), mais les paragraphes 3 et 4 n'ont quasiment rien
produit (Fantine, Cosette, les Thénardier absents). La sortie contient des
entités absurdes (« passeport jaune », « argentie de l'évêque Myriel » avec
faute) et des noms de relations incohérents (« déclare avoir donné au voleure,
en y ajoutant autre chose »). Hypothèse : le décodage contraint par JSON Schema
d'Ollama dégrade fortement ce modèle. À écarter pour cette tâche.

## Limites du bench

- **Un seul run par modèle** : pas de mesure de variance ; un écart d'une
  entité ou relation entre deux modèles proches n'est pas significatif.
- **Un seul texte, court et canonique** : Les Misérables est massivement
  présent dans les données d'entraînement ; les scores sur un texte inédit
  seraient probablement plus bas, et l'écart entre modèles peut se resserrer
  ou s'amplifier.
- **Chunks de 700 caractères** : bien plus petits que les 8 000 par défaut ;
  les temps ne sont pas directement extrapolables (le coût par chunk croît
  avec sa taille).
- Le scoring des relations est indulgent (tout lien entre les deux bonnes
  entités compte) et celui des entités dépend des variantes de noms prévues.

## Recommandations

**Choix de modèle**
- Qualité maximale en local : `gpt-oss:120b`.
- Roman entier en temps raisonnable : `gpt-oss:20b` (~3 h estimées pour
  ~500 chunks de 8 000 caractères, contre ~8 h pour le 120b — estimation
  grossière, à recaler sur de vrais chunks).
- Prototypage rapide : `qwen2.5:7b-instruct`.
- À éviter : `mistral-small3.2:24b` pour cette tâche.

**Durcissement du pipeline** (issues des défauts observés sur le 120b)
1. Ignorer les entités au nom vide et les relations sans source ou cible.
2. Normaliser l'Unicode dans `normalize()` (NFKC + unification des tirets
   U+2010–U+2015) pour fusionner « Montreuil-sur-Mer » / « Montreuil‑sur‑Mer ».
3. Étendre la résolution aux préfixes de civilité/titre (« inspecteur Javert »
   vs « Javert ») — avec prudence, c'est une heuristique.
4. Paralléliser les appels LLM par lots pour les longs textes.

## Reproduction

```bash
# Prérequis : Ollama local avec les modèles installés, venv du repo
.venv/bin/python benchmarks/bench.py                    # liste par défaut
.venv/bin/python benchmarks/bench.py gpt-oss:20b        # un seul modèle
```

Les résultats sont écrits dans `benchmarks/results/<date>/` : un
`bench_results.json` consolidé (scores) et un graphe JSON complet par modèle.
